    <h2 className="section-title">Q & A</h2>
    
    <div className="tech-card">
      <b>Q1：物理机环境，如果内存资源不足也会被 killed 吗？相对来说容器这方面有什么规避方案吗？</b>
      <p style={{ marginTop: '10px' }}>A：在物理机上在 <code>/etc/sysctl.conf</code> 中配置 <code>vm.overcommit_memory=2</code>（以及一些其它参数）之后，不会发生数据库被 OOM 给 killed 掉的情况。这是，当内存不足时，执行的 SQL 会因为申请不到新内存而直接报错，符合用户的预期。而如果使用容器的内存限制，则无法达到这个效果。目前这个问题在容器中没有太好的解决办法。</p>
    </div>

    <div className="tech-card">
      <b>Q2：IvorySQL 有没有自带的高可用架构？</b>
      <p style={{ marginTop: '10px' }}>A：IvorySQL 与 Patroni 完美适配，Patroni 作为支持 PG 的高可用工具，十分完备且社区活跃，可以通过 Patroni 进行 IvorySQL 高可用集群搭建。</p>
    </div>

    <div className="tech-card">
      <b>Q3：IvorySQL 在兼容 Oracle 的时候对 Oracle 有版本要求吗？</b>
      <p style={{ marginTop: '10px' }}>A：完全可以将 IvorySQL 作为 Oracle 数据库进行使用，与 Oracle 自身并无关联。</p>
    </div>

    <div className="tech-card">
      <b>Q4：兼容模式与 database 模式的区别以及这两者不同搭配值的行为差异？</b>
      <p style={{ marginTop: '10px' }}>A：database 模式决定了初始化时，初始化的是 Oracle 兼容的 PG 数据库还是原生的 PG 数据库，一旦初始化就无法进行修改。原生的 PG 数据库无法支持 Oracle 兼容特性。</p>
    </div>

    <div className="tech-card">
      <b>Q5：IvorySQL operator 支持垂直扩缩容吗？支持程度是怎么样的？</b>
      <p style={{ marginTop: '10px' }}>A：支持数据库 pod 的 CPU、Memory 垂直扩缩容，并且用户可以随时进行调整。</p>
