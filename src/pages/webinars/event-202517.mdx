    <div className="tech-card">
      <b>Q1: Will switching to compatibility mode online affect data? For example, Oracle only has NULL while PG has empty strings.</b>
      <p style={{ marginTop: '10px' }}>A: Switching IvorySQL to Oracle compatibility mode primarily affects syntax, functions, and data type behavior, but it does not modify existing data: records that originally distinguished between '' and NULL in the database will remain as they are.</p>
    </div>

    <div className="tech-card">
      <b>Q2: Is v5.1 live yet? To what extent has the new version achieved compatibility for specific Oracle syntax (like PL/SQL packages or certain data types)? Are there known limitations or alternatives?</b>
      <p style={{ marginTop: '10px' }}>A: IvorySQL 5.1 is now live. IvorySQL's core procedural language is PL/iSQL, which introduces Oracle-style Packages, stored procedures, functions, and nested sub-functions, supporting common data types. Current known limitations: Packages only support partial features and are still under continuous development. Alternative: Use custom functions for implementation.</p>
    </div>

    <div className="tech-card">
      <b>Q3: Do nested sub-functions support multiple levels of nesting? Is there a limit on nesting levels?</b>
      <p style={{ marginTop: '10px' }}>A: Yes, multiple levels of nesting are supported. To prevent infinite recursion and resource exhaustion, the nesting limit is set to 200 levels.</p>
    </div>

    <div className="tech-card">
      <b>Q4: Are there migration tools for Oracle stored procedure migration? How can the correctness after migration be verified?</b>
      <div style={{ marginTop: '10px' }}>
        A: Open-source version: Only provides table and data migration, usually not supporting stored procedure migration. Highgo's commercial version supports stored procedure migration, typically verified through five levels:
        <ul>
          <li>Syntax creation success: Confirm no syntax errors;</li>
          <li>Data-less logic verification: Execute logic tests in a test environment;</li>
          <li>Data-driven business verification: Use simulated/desensitized production data to verify business correctness;</li>
          <li>Application verification: Verify functional completeness through application calls;</li>
          <li>Regression test verification: Verification through user regression testing systems.</li>
        </ul>
      </div>
